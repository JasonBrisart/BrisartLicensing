# Implementation Requests

Implementation requests allow subscribers and ecosystem participants to suggest future capabilities for Brisart software projects.

The Brisart ecosystem is actively developed around local-first, auditable, maintainable, and research-oriented software.

Implementation requests help identify useful future directions.

---

## Purpose

Implementation requests are used to suggest:

- New features
- New workflows
- New tools
- New research capabilities
- New preservation capabilities
- New identity capabilities
- New development utilities
- New ecosystem integrations

---

## Example Request Areas

Implementation requests may involve areas such as:

- Local identity records
- Offline authentication workflows
- Physical token verification
- Credential workflows
- Research archive features
- Framework execution tools
- Preservation utilities
- File verification tools
- Developer automation tools
- Report generation
- Local-first data management
- Air-gapped environment workflows

---

## Request Review

Implementation requests may be reviewed based on:

- Alignment with the Brisart ecosystem
- Usefulness to current or future Brisart projects
- Relevance to local-first or offline workflows
- Relevance to research, archival, identity, or preservation environments
- Maintainability
- Simplicity
- Long-term value

---

## No Guarantee

Submitting an implementation request does not guarantee that the request will be implemented.

Brisart may accept, reject, modify, defer, or reinterpret implementation ideas based on ecosystem priorities and development direction.

---

## Product Direction

Implementation requests are intended to influence future development.

They are not support tickets, bug tickets, consulting requests, or managed-service requests.

Brisart develops products.

Implementation requests help identify what may be worth building next.

---

## Ecosystem Fit

Requests that fit the Brisart ecosystem are more likely to be considered.

The ecosystem generally prioritizes:

- Local-first operation
- Source availability
- Auditability
- Long-term maintainability
- Dependency-conscious development
- Research usage
- Archival usage
- Preservation workflows
- Offline environments
- Understandable code
